01

Visión y objetivo

Gestión 360 iA es una plataforma SaaS multi-tenant modular con inteligencia artificial integrada como capa transversal, orientada a pymes argentinas de distintos rubros, con proyección regional.

El objetivo a 2 años es posicionarse como una de las consultoras e integraciones IA más importantes de Argentina.

◆
Modelo de negocio
El cliente descarga la app de su rubro, completa el onboarding, y obtiene acceso automático según el plan adquirido. Sin aprobación manual. Escala a miles de usuarios simultáneos.
✦
Diferenciador principal — Skills IA
Sugerencias proactivas y reactivas generadas por IA específicas para cada rubro. No existe equivalente en Argentina ni LATAM. Es el principal driver de valor y retención.

6 pilares fundamentales

Antes de escribir una línea de código en cualquier app nueva, estos 6 pilares deben estar definidos y resueltos. Son no negociables.

Pilar 01
Arquitectura de carpetas
Cómo está organizado el código. Qué va en qué carpeta, dónde viven las rutas, los componentes y la lógica de negocio. Define si el proyecto es mantenible a escala.
Ver sección 08 →
Pilar 02
Autenticación y sesiones
Quién puede acceder y cómo se gestiona la sesión. NextAuth independiente por app. JWT en httpOnly cookies. Cada app su propio secret — nunca compartido.
Ver sección 10 →
Pilar 03
Autorización y roles (RBAC)
Auth dice "sos Pablo". RBAC dice "Pablo puede ver esto pero no aquello". Middleware en cada ruta. Roles diferenciados por app.
middleware.js + permisos_modulo
Pilar 04
Aislamiento multi-tenant
Ningún tenant puede ver datos de otro. Una DB por tenant. El dbName siempre viene del token de sesión — nunca del body o query string.
Ver sección 11 →
Pilar 05
Gestión de secretos
API keys, passwords, tokens. Nunca en el código ni en el repo. Siempre en variables de entorno del VPS en Easypanel. El .env nunca se commitea.
Easypanel → Entorno
Pilar 06
Observabilidad
Logs, auditoría, monitoreo. Saber qué pasó, cuándo y quién lo hizo. Winston para logs de app, tabla sesiones_log en DB, health checks en Easypanel.
sesiones_log + Winston + PM2

⚠
3 reglas de arquitectura — no negociables
Regla 1: El tenant siempre viene del token de sesión — nunca del query string ni del body.
Regla 2: Nunca conectar a la DB del tenant sin verificar que el dbName existe y está activo en la DB core.

Base de datos y tablas

Arquitectura multi-tenant con base de datos separada por cliente. Hay dos universos: la DB core del sistema y las DBs individuales de cada tenant.

◈
Regla de conexión
Toda API que necesite datos del tenant usa lib/tenant-db.js con el dbName que viene del token de sesión. Nunca se acepta dbName del body o query string. Nunca se conecta a la DB de otro tenant.

Principios de trabajo

  • Avanzamos paso a paso — un componente o módulo a la vez
  • Cada entrega debe ser funcional y desplegable
  • Velocidad de iteración sobre perfección inicial
  • Ante cada decisión técnica importante: opciones con pros y contras antes de implementar
  • Código limpio, comentado y escalable desde el día 1
  • Todo el código es generado por IA y subido via GitHub web o file manager de Ferozo
  • Despliegues semanales los miércoles
  • Las API keys y credenciales SIEMPRE en variables de entorno del VPS — nunca en el repo
  • Auth independiente por app — nunca compartir NEXTAUTH_SECRET entre admin y portal
  • DB separada por tenant desde el día 1 — no hay marcha atrás en esto
  • El tenant siempre viene del token — nunca del body o query string
  • Cuando algo no esté claro, se documenta antes de implementar
